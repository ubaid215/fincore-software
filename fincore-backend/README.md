## 1. Architecture Overview

```
                        ┌─────────────────────────────────────┐
                        │        React 18 + Vite Frontend       │
                        └──────────────────┬──────────────────┘
                                           │ REST (OpenAPI 3.1)
                        ┌──────────────────▼──────────────────┐
                        │     NestJS API (v1)   port 3000      │
                        │  JWT RS256 · RBAC · OpenAPI 3.1      │
                        └────┬──────┬──────┬──────────────────┘
                             │      │      │
                    ┌────────▼─┐  ┌─▼────┐ ┌──▼──────┐
                    │PostgreSQL│  │Redis │ │BullMQ   │
                    │16 (RDS)  │  │7     │ │Job Queue│
                    └──────────┘  └──────┘ └─────────┘
                             │
             ┌───────────────┴─────────────┐
             │       External Services      │
             ├─────────────────────────────┤
             │  AWS S3 (receipts, PDFs)    │
             │  Keycloak (OAuth2/MFA)      │
             │  Open Exchange Rates (FX)   │
             │  SMTP (Nodemailer)          │
             └─────────────────────────────┘

             ── Post-MVP (Phase 2) ──────────
             │  HashiCorp Vault (PKI/sign)  │
             │  Sectigo TSA (timestamps)    │
             │  JazzCash / HBL Konnect      │
             └──────────────────────────────┘
```

**Key design decisions:**

- **`organizationId` on every table** — multi-tenant isolation; Row-Level Security enforced at application layer via Prisma middleware
- **Double-entry enforced at service layer**: `SUM(debits) = SUM(credits)` validated before every DB write
- **Decimal precision**: `DECIMAL(19,4)` for all monetary values — no floating-point arithmetic
- **Feature flags via Redis** with 60 s TTL — plan entitlements cached, not re-queried per request
- **Manual payment abstraction layer** — `IPaymentGateway` interface ready for JazzCash plug-in (Phase 2)
- **Split Prisma schema** — one `.prisma` file per domain; merged at `prisma generate` time via `prismaSchemaFolder` preview feature
- **Digital Signatures → Post-MVP** — HashiCorp Vault PKI, PAdES-B-LT, RFC 3161 timestamp deferred to Phase 2

## 3. Prisma Schema Strategy

The single `schema.prisma` file is replaced by a **domain-split schema folder**. Each domain owns its own `.prisma` file. Prisma merges them at `generate` and `migrate` time via the `prismaSchemaFolder` preview feature (available since Prisma 5.15).

### Configuration (`prisma/schema/base.prisma`)

```prisma
generator client {
  provider        = "prisma-client-js"
  previewFeatures = ["prismaSchemaFolder"]
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}
```

The `generator` and `datasource` blocks live **only** in `base.prisma`. Every other file contains only models and enums for its domain.

### How it works

```bash
# Prisma automatically discovers all *.prisma files in prisma/schema/
npx prisma generate          # Merges all files → generates single PrismaClient
npx prisma migrate dev       # Merges all files → creates one migration SQL
npx prisma migrate deploy    # Production migration (same merge, no prompt)
```

### Domain file ownership

| File | Models | Owner |
|------|--------|-------|
| `base.prisma` | generator, datasource | DevOps / Lead |
| `auth.prisma` | User, RefreshToken, Invite | Auth team |
| `workspace.prisma` | Organization, UserOrganization | Workspace team |
| `chart-of-accounts.prisma` | Account, FiscalPeriod | GL team |
| `general-ledger.prisma` | JournalEntry, JournalLine | GL team |
| `invoicing.prisma` | Invoice, InvoiceLineItem, InvoicePayment | Invoicing team |
| `expenses.prisma` | Expense, ExpenseLine, Receipt | Expense team |
| `bank-reconciliation.prisma` | BankStatement, BankTransaction | Recon team |
| `subscriptions.prisma` | Plan, Subscription | Billing team |
| `manual-payments.prisma` | ManualPayment | Billing team |
| `audit.prisma` | AuditLog | Cross-cutting |

### Cross-domain relations

When a model in one file references a model in another, Prisma resolves it at generation time — no special syntax required:

```prisma
// invoicing.prisma — references User from auth.prisma and Organization from workspace.prisma
model Invoice {
  id             String       @id @default(uuid())
  organizationId String                            // FK → Organization (workspace.prisma)
  organization   Organization @relation(...)
  // Prisma resolves Organization at generate time — no import needed
}
```

### Rules

1. **Only `base.prisma` contains `generator` and `datasource`** — never repeat them.
2. **One domain per file** — models that always migrate together stay together.
3. **No circular relations across more than 2 files** — keep the dependency graph a DAG.
4. **Migrations are still a single SQL file** — Prisma merges all schemas before diffing.

## 6. Testing Architecture

### 6.1 Testing Philosophy

FinCore uses a **three-layer testing pyramid**:

```
        ┌─────────────────────────────┐
        │   E2E  (Jest + Supertest)   │  ~100 scenarios — full HTTP stack
        │   test/e2e/*.e2e-spec.ts    │  Real NestJS app + real PG + real Redis
        ├─────────────────────────────┤
        │  Integration  (Jest+Prisma) │  ~30 scenarios — service + real DB, no HTTP
        │  test/integration/*.spec.ts │  Validates DB constraints & cross-service flow
        ├─────────────────────────────┤
        │    Unit  (Jest, all mocked) │  ~200 scenarios — fastest, runs in CI always
        │  src/**/*.spec.ts           │  PrismaService mocked, external deps mocked
        └─────────────────────────────┘
```

**Rules:**
1. Every **service method** has unit tests. Controllers have unit tests for guard/pipe logic.
2. Every **cross-service flow** (e.g. expense approval → GL post) has an integration test.
3. Every **API endpoint** has at least one E2E test covering the happy path + primary error case.
4. **No test touches another test's data** — each spec file cleans the DB in `beforeEach` / `afterAll`.
5. **Mocks are explicit, never `jest.mock()`-all** — mock only what you need, assert mock calls.

```typescript
import { INestApplication, ValidationPipe } from '@nestjs/common';
import { Test } from '@nestjs/testing';
import * as supertest from 'supertest';
import { AppModule } from '../../src/app.module';

let app: INestApplication;

export async function createTestApp(): Promise<INestApplication> {
  if (app) return app;  // Singleton — reuse across spec files in the same run

  const module = await Test.createTestingModule({ imports: [AppModule] }).compile();
  app = module.createNestApplication();
  app.useGlobalPipes(new ValidationPipe({ whitelist: true, transform: true }));
  await app.init();
  return app;
}

export function api(app: INestApplication) {
  return supertest(app.getHttpServer());
}
```

**Auth helper (`test/helpers/auth.helper.ts`):**

```typescript
// Mints a test JWT without going through the login endpoint — keeps E2E fast
export async function getTestToken(
  app: INestApplication,
  role: UserRole = UserRole.OWNER,
): Promise<{ token: string; orgId: string; userId: string }> {
  const prisma = app.get(PrismaService);
  const user = await prisma.user.create({ data: testUserFactory(role) });
  const org  = await prisma.organization.create({ data: testOrgFactory() });
  await prisma.userOrganization.create({ data: { userId: user.id, organizationId: org.id, role } });

  const jwtService = app.get(JwtService);
  const token = jwtService.sign({ sub: user.id, email: user.email });
  return { token, orgId: org.id, userId: user.id };
}
```

```typescript
// test/e2e/invoicing.e2e-spec.ts
import * as request from 'supertest';
import { createTestApp, api } from '../helpers/app.helper';
import { getTestToken } from '../helpers/auth.helper';
import { cleanDb }      from '../helpers/db.helper';

describe('Invoicing (e2e)', () => {
  let app: INestApplication;
  let token: string;
  let orgId: string;

  beforeAll(async () => {
    app = await createTestApp();
    ({ token, orgId } = await getTestToken(app, UserRole.ADMIN));
  });

  beforeEach(() => cleanDb(app));    // Wipe relevant tables between tests
  afterAll(()  => app.close());

  const auth = () => ({ Authorization: `Bearer ${token}`, 'X-Organization-Id': orgId });

  // ── Happy path ─────────────────────────────────────────────────────────────
  it('POST /v1/invoices → 201 with INV- number', async () => {
    const { body } = await api(app)
      .post('/v1/invoices')
      .set(auth())
      .send(validInvoicePayload)
      .expect(201);

    expect(body.data.invoiceNumber).toMatch(/^INV-\d{4}-\d{6}$/);
    expect(body.data.status).toBe('DRAFT');
    expect(body.data.totalAmount).toBe(50000);
  });

  // ── Validation ─────────────────────────────────────────────────────────────
  it('POST /v1/invoices → 400 with empty lineItems', () =>
    api(app).post('/v1/invoices').set(auth())
      .send({ ...validInvoicePayload, lineItems: [] }).expect(400));

  it('POST /v1/invoices → 401 without token', () =>
    api(app).post('/v1/invoices').send(validInvoicePayload).expect(401));

  it('POST /v1/invoices → 403 for VIEWER role', async () => {
    const { token: viewerToken, orgId: vOrgId } = await getTestToken(app, UserRole.VIEWER);
    await api(app).post('/v1/invoices')
      .set({ Authorization: `Bearer ${viewerToken}`, 'X-Organization-Id': vOrgId })
      .send(validInvoicePayload).expect(403);
  });

  // ── State machine ──────────────────────────────────────────────────────────
  it('Full lifecycle: DRAFT → SENT → PAID', async () => {
    const { body: { data: invoice } } = await api(app)
      .post('/v1/invoices').set(auth()).send(validInvoicePayload).expect(201);

    await api(app).patch(`/v1/invoices/${invoice.id}/send`).set(auth()).expect(200);

    const { body: paid } = await api(app)
      .post(`/v1/invoices/${invoice.id}/payments`).set(auth())
      .send({ amount: 50000, currency: 'PKR', method: 'bank_transfer', paidAt: '2025-03-15' })
      .expect(201);

    expect(paid.data.status).toBe('PAID');
  });

  it('PATCH /v1/invoices/:id/void → 409 on PAID invoice', async () => {
    // Create + pay invoice, then attempt void
    const id = await createAndPayInvoice(app, auth());
    await api(app).patch(`/v1/invoices/${id}/void`).set(auth()).expect(409);
  });
});
```

## 7. Running the Project

### Prerequisites

- Node.js 22+
- Docker + Docker Compose
- k6 (optional — for load tests only)

### Quick Start

```bash
# 1. Clone and install
git clone https://github.com/your-org/fincore-backend.git
cd fincore-backend
npm install

# 2. Start local infrastructure
docker-compose -f docker/docker-compose.yml up -d
# Starts: PostgreSQL 16, Redis 7, MailHog (email UI at localhost:8025)

# 3. Configure environment
cp .env.example .env.local
# Edit: DATABASE_URL, REDIS_HOST, JWT_PRIVATE_KEY_PATH, etc.

# 4. Generate RSA key pair for JWT
bash scripts/generate-rsa-keys.sh
# Creates: keys/private.pem and keys/public.pem

# 5. Generate Prisma client from split schema
npm run db:generate

# 6. Run migrations (merges all prisma/schema/*.prisma files)
npm run db:migrate

# 7. Seed reference data (plans, GAAP/IFRS CoA templates)
npm run db:seed

# 8. Start development server
npm run start:dev

# API:    http://localhost:3000/v1
# Docs:   http://localhost:3000/docs   (Swagger UI)
# Health: http://localhost:3000/health
# Mail:   http://localhost:8025        (MailHog)
```

### Database Scripts

```bash
npm run db:generate          # Regenerate Prisma client (run after any schema change)
npm run db:migrate           # Create + apply new migration (dev)
npm run db:migrate:prod      # Apply pending migrations (production — no prompt)
npm run db:seed              # Run all seed files via prisma/seeds/index.ts
npm run db:reset             # Drop + recreate + migrate + seed (destructive — dev only)
npm run db:studio            # Open Prisma Studio (visual DB browser)
```
